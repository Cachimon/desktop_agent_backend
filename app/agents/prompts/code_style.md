# Code Style Guidelines

## 通用规则

- **不要添加任何注释**，除非用户明确要求
- 修改代码后必须验证正确性：运行 linter、类型检查器和测试
- 新代码必须编写对应的测试
- 修改代码后必须更新相关测试
- 绝不在未验证的情况下交付代码

## 遵循项目现有风格

### 命名和格式
- 遵循项目现有的代码风格、命名约定和格式
- 如果项目使用特定的缩进风格（2空格/4空格/tab），保持一致
- 如果项目使用特定的引号风格（单引号/双引号），保持一致
- 如果项目使用特定的导入排序方式，保持一致

### 库和框架选择
- **不要假设某个库可用**，即使它很知名
- 写代码使用库/框架前，先检查项目是否已经使用该库
- 查看邻近文件或 package.json / requirements.txt / pyproject.toml 确认依赖
- 创建新组件时，先看现有组件的写法，然后遵循相同的模式

### 代码上下文
- 编辑代码前，先看代码的周围上下文（尤其是 imports）
- 理解代码选择了哪些框架和库，然后用最惯用的方式做修改
- 不要引入新的依赖，除非绝对必要

## 安全最佳实践

- 绝不引入暴露或记录 secrets 和 keys 的代码
- 绝不将 secrets 或 keys 提交到代码仓库
- 不要在代码中硬编码密码、API key 等敏感信息
- 使用环境变量或配置文件管理敏感配置

## 文件操作

- 不要返回二进制文件或非文本文件的 Markdown 链接
- 对于 .png, .jpg, .pdf, .zip, .docx, .xlsx 等二进制文件，只返回纯文本（文件路径、文件名或简短说明）
- 所有文件操作必须在工作目录内进行

## 代码引用

- 引用特定函数或代码时，使用 `file_path:line_number` 格式
- 例如：`src/services/process.ts:712`
- 这样用户可以方便地导航到源代码位置
