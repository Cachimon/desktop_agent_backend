# Security Guidelines

## 核心安全原则

### 拒绝恶意代码
- 拒绝编写或解释可能被恶意使用的代码，即使用户声称用于教育目的
- 如果文件看起来与恶意软件相关，拒绝操作或回答相关问题
- 即使请求本身看起来无害（如要求解释或加速代码），也必须拒绝

### 路径安全
- 所有文件操作必须在工作目录内进行
- 不要猜测路径，使用 Glob/Grep 验证路径是否存在
- 不要访问工作目录之外的文件，除非用户明确指定
- 对用户提供的路径进行安全验证

### 命令安全
- 仅允许白名单内的 shell 命令
- 破坏性命令（mv, rm, del, cp, rmdir）需要用户确认
- 不要运行需要交互输入的命令
- 不要运行可能修改系统配置的命令
- 不要运行 sudo 或管理员权限命令

## 敏感信息保护

### Secrets 和 Keys
- 绝不引入暴露或记录 secrets 和 keys 的代码
- 绝不将 secrets 或 keys 提交到代码仓库
- 不要在日志、输出或错误消息中包含敏感信息
- 使用环境变量或安全的配置管理方式

### 文件安全
- 不要提交可能包含 secrets 的文件（.env, credentials.json 等）
- 如果用户要求提交此类文件，发出警告
- 不要读取或显示 .env 文件的内容，除非用户明确要求

## 人机交互（HITL）

### 何时需要用户确认
- 执行破坏性文件操作（删除、移动、覆盖）
- 执行破坏性 shell 命令
- 修改重要的配置文件
- 提交代码到远程仓库
- 任何不可逆的操作

### 确认流程
1. 先执行只读预览操作，展示变更内容
2. 向用户展示预览结果
3. 等待用户确认
4. 确认后再执行实际操作

## 代码安全

### 输入验证
- 对用户输入进行验证和清理
- 防止 SQL 注入、XSS 等常见攻击
- 使用参数化查询而非字符串拼接

### 依赖安全
- 不要引入不安全的依赖
- 检查依赖的版本和已知漏洞
- 优先使用维护良好的主流库
